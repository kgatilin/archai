# archai как retrieval-сервис для оркестратора (loom)

> Plan / design note. Граница: **archai = знание о коде** (граф + эмбеддинги +
> поиск), **loom (внешний оркестратор) = что делать дальше** (routing, циклы,
> LLM-судья). archai не знает про оркестрацию; loom не парсит код и не держит
> вектора. Документ фиксирует разрез и что для него доделать в archai.

---

## 0. Зачем

archai уже парсит проект в AST-граф (символы func/struct/iface/type/const/var,
рёбра, `Doc`/докстринг на ноду) и отдаёт его самодостаточным JSON (`uigraph`).
Следующий шаг — превратить archai в **stateful retrieval-сервис над кодовой
базой**: на каждую ноду эмбеддинг + ANN + BM25, поиск (semantic + lexical) и
обход по графу за одним API. Тогда любой потребитель (агент-оркестратор loom,
IDE, MCP-клиент) получает «что релевантно в коде» одним вызовом, а сам логику
«что делать с этим» держит у себя.

## 1. Граница archai ↔ оркестратор

| archai (этот репо) | оркестратор (loom, отдельно) |
|---|---|
| парсит проект → AST-граф (символы, рёбра, docstring) | **не** парсит код, **не** держит эмбеддинги |
| эмбеддинги на ноду + ANN (HNSW) + inverted BM25 | решает **когда** искать и **с каким** запросом |
| freshness: индекс хранится в проекте, mtime/hash, `refresh` | LLM-судья: «достаточно / refine / новый под-вопрос» |
| fused search (dense+BM25+RRF) + graph-expand **внутри себя** | spawn read/impl-агентов, join, план, бюджеты, retry |
| отдаёт ранжированный + расширенный по графу набор нод | зовёт archai как tool, по ответу маршрутизирует |

Тот же раздел, что в retrieval-vs-world-model: **archai = «что релевантно»
(knowledge), оркестратор = «что делать дальше» (routing).**

## 2. Про скорость (снять недоразумение)

Поиск по всему проекту — **самая дешёвая** часть, не медленная:
- **BM25** не сканирует файлы — работает через **inverted index** (терм → ноды);
  трогаются только ноды с термами запроса → top-K за миллисекунды на 50k символов.
- **dense** в проде — не линейный перебор, а **ANN (HNSW)** → тоже миллисекунды.

Медленное — LLM-судья и чтение файлов (на стороне оркестратора), не retrieval.

## 3. API archai (минимум)

```
POST /search       {query, k, filters}        → [{node_id, kind, file, doc, score, snippet}]
POST /search_graph {query, k, hops}           → search + авто-expand, отдаёт подграф
POST /expand       {node_ids, hops, edges}    → соседние ноды по рёбрам графа
GET  /node/{id}                               → полное тело + рёбра ноды
POST /refresh                                 → переиндексировать изменённые файлы
```

`/search_graph` — ключевой: archai сам делает retrieve → graph-expand и
возвращает готовый контекст-подграф; потребитель по графу руками не ходит.

Поверх уже существующих HTTP/MCP-поверхностей archai — это новые ручки, не новый
сервер.

## 4. Что доделать в archai

Граф, HTTP/MCP и докстринги уже есть. Не хватает:

1. **Эмбеддинги на ноду + ANN.** Энкодер для текста `[сигнатура + docstring]`
   на символ; ANN-индекс (HNSW) для cosine top-K. (Проверить, есть ли уже частичная
   поддержка эмбеддингов; если нет — добавляется поверх готового графа.)
2. **BM25 inverted index** по тому же тексту + идентификаторам — lexical-проход.
3. **Fusion (RRF)** двух списков в `/search`; **graph-expand** в `/search_graph`.
4. **Freshness-менеджмент.** Индекс сохраняется в проекте (рядом с конфигом
   archai), на ноду — hash/mtime источника; `refresh` (или file-watch)
   переиндексирует только изменённые файлы. archai становится держателем
   состояния «актуальности» проекта.
5. (опционально) **rerank** — cross-encoder на паре `(query, chunk)` для точного
   финального ранжирования; старт без него, на cosine+BM25+RRF.

## 5. Чанкинг (для эмбеддингов)

Резать по **AST-границам**, не по символам/токенам (структурные границы важнее
размера — бенчмарки 2025, cAST):
- 1 чанк = 1 синтаксическая единица (функция/метод/тип), бюджет ~512 токенов;
- большой символ > бюджета → бить по AST-подблокам, **сигнатуру заголовком** на
  каждый кусок;
- мелкие → паковать соседние до бюджета;
- к чанку приклеивать enclosing-контекст (package/imports + имя типа).
Для Go — нативно через `go/ast` (точнее tree-sitter); `ast.IsExported` бесплатно
делит public/internal (возможен two-tier поиск: сначала public-сигнатуры, потом
internal внутри найденного).

## 6. Сторона оркестратора (loom) — вне этого репо, для контекста

Субстрат loom менять не надо. Нужен тонкий tier-3 плагин (НЕ в этом репо):
- tool-плагин, зовущий archai по HTTP, declaration с лейблом **`[:archai_read]`**
  (зависимость на archai явная — это набор «archai-tools», не абстрактный
  «code_read»);
- reaction-петля research/судьи: `query → archai /search_graph → LLM-judge →
  enough?/refine` (self-loop, бюджет итераций через cone);
- endpoint archai в ambient context (`context.code.endpoint`).

Статус: пока **не реализуем** — фиксируем границу и контракт.

## 7. Поток (end-to-end, для наглядности)

```
оркестратор: запрос (таска / под-вопрос)
   └─▶ archai /search_graph {query,k,hops}
          dense(ANN) + BM25(inv) ─▶ RRF fusion ─▶ graph-expand ─▶ (rerank?) ─▶ подграф нод
   ◀── ранжированные ноды + контекст
оркестратор: read top-N ─▶ LLM-судья «достаточно?»
          ├─ да    ─▶ выход (контекст собран)
          └─ нет   ─▶ новый под-вопрос ─▶ archai /search_graph (петля, лимит итераций)
```

retrieval/expand/rank — целиком в archai; «когда/с чем/сколько раз/что дальше» —
целиком в оркестраторе.
